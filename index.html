Essencial de Git e GitHub

Conceitos de Git e GitHub

O que é git?

Git é um sistema de controle de versão distribuído. Ele registra o histórico de alterações do seu projeto e permite trabalhar em paralelo com outras pessoas.

  • commit — ponto no tempo com suas mudanças e uma mensagem.
  • branch — linha de desenvolvimento isolada (ex.: main, feature/login).
  • merge — une o histórico de duas branches.
  • rebase — reaplica seus commits sobre outra base, criando histórico linear.
  • remote — repositório hospedado (ex.: GitHub) para compartilhar código.

O que é GitHub?

GitHub é uma plataforma de hospedagem de repositórios git com colaboração: issues, pull requests, revisões, CI/CD e gerenciamento de permissões.

  • fork — cópia do repo na sua conta para contribuir com projetos de terceiros.
  • pull request (PR) — proposta de mudança de uma branch para outra, com revisão.
  • issue — tarefa/bug/idéia rastreável.
  • actions — automações (testes, build, deploy) a cada push/PR.

Dica: Commits pequenos e bem descritos facilitam revisão, git blame e revert.

Fluxo recomendado (Git/GitHub)

  1. 1) Clonar e configurar 
    git clone <url-do-repo>
cd <pasta>
# Defina seu usuário (uma vez por máquina)
git config --global user.name "Seu Nome"
git config --global user.email "voce@example.com"
  2. 2) Atualizar a base local 
    git checkout main
# traz as últimas mudanças do remoto
git pull --ff-only origin main

    Use --ff-only para evitar merges desnecessários no histórico principal.

  3. 3) Criar uma branch por tarefa 
    # nome curto e descritivo
git checkout -b feature/login
  4. 4) Trabalhar e comitar 
    # ver status/alterações
git status
git add <arquivos>
# mensagem no imperativo, curta + contexto
git commit -m "cria formulário de login com validação"
  5. 5) Sincronizar frequentemente 
    # rebase mantém histórico linear
git fetch origin
git rebase origin/main

    Caso prefira, git merge origin/main também funciona, porém cria commits de merge.

  6. 6) Enviar e abrir PR 
    git push -u origin feature/login
# No GitHub: abrir Pull Request para a branch main

    Garanta testes passando e descrição clara no PR. Solicite revisão.

  7. 7) Revisão, ajustes e merge

    Após aprovarem, faça merge (ou squash merge para condensar commits) e delete a branch.

Como resolver conflitos

Conflitos acontecem quando duas mudanças tocam as mesmas linhas em arquivos diferentes. Passo a passo:

1) Reproduzir o conflito com segurança

# atualize sua branch com a base mais recente
git fetch origin
git rebase origin/main  # ou: git merge origin/main

2) Ver o que conflitou

git status   # lista arquivos em conflito
# Abra os arquivos marcados com <<<<<<< e >>>>>>>

Marcadores típicos num arquivo:

<<<<<<< HEAD
sua versão
=======
versão da base (origin/main)
>>>>>>> origin/main

3) Editar manualmente e marcar como resolvido

# edite e mantenha a versão correta (ou combine as duas)
# depois:
git add caminho/arquivo.ext

4) Continuar o fluxo

# se estiver num rebase:
git rebase --continue
# se estiver num merge:
git commit   # cria o commit de merge

5) Testar e enviar

npm test      # ou seus testes/build
git push

Se algo der errado:

  • git rebase --abort — cancela o rebase e volta ao estado anterior.
  • git merge --abort — cancela o merge em andamento.
  • git restore --source=HEAD -- <arquivo> — descarta alterações no arquivo.

Ferramentas que ajudam

  • VS Code: painel de Merge com botões (Accept Current/Incoming/Both).
  • git mergetool: integra com difftools (Meld, KDiff3, Beyond Compare).
  • Proteja a main no GitHub com PRs + revisões obrigatórias para reduzir conflitos.

Boas práticas para evitar conflitos:

  • Branches curtas e focadas; sincronize com a base diariamente.
  • Arquitetura modular: reduza arquivos muito grandes com responsabilidades únicas.
  • Padronize formatação (Prettier/EditorConfig) e convenções de mensagem de commit.
